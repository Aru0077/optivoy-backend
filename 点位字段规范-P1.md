# 点位字段规范（P1 冻结版）

版本：`v1.1`  
日期：`2026-04-01`  
状态：`Backend Phase 4 已同步`

## 1. 目标与范围

本规范用于冻结 Trip Planner 的字段字典，覆盖以下类型：

- `spot`
- `shopping`
- `restaurant`
- `hotel`
- `airport`

本阶段以“字段与枚举冻结”为主线，机票深链保持现状；当前仓库已完成 backend 到 optimizer 的基础契约接入，并开始透传现实约束字段。

## 2. 枚举冻结

### 2.1 点位类型

- `spot`：景点
- `shopping`：购物
- `restaurant`：饭店
- `hotel`：酒店
- `airport`：机场

### 2.2 强度模式（PaceMode）

- `light`：`10:00-18:00`，有效 `480` 分钟
- `standard`：`09:00-20:00`，有效 `660` 分钟
- `compact`：`08:00-21:00`，有效 `780` 分钟

### 2.3 酒店策略（HotelMode）

- `single`：全程单酒店
- `multi`：多酒店最优方案

### 2.4 午餐策略（MealPolicy）

- `auto`：系统自动处理午餐约束（默认）
- `off`：不强制午餐约束

## 5. 求解输入契约（冻结草案）

### 5.1 用户可选点位范围

- 仅 `spot/shopping/restaurant` 可由用户手动选择
- `hotel/airport` 作为求解约束输入，不放在用户点选列表

### 5.2 最小求解输入（后续接 optimizer 使用）

- `city`
- `province`
- `arrivalDateTime`
- `selectedPointIds`（无序）
- `paceMode`（`light/standard/compact`）
- `hotelMode`（`single/multi`）
- `mealPolicy`（`auto/off`）
- `arrivalAirportCode`（可选）
- `departureAirportCode`（可选）

当前 backend 还会向 optimizer 透传以下结构化数据：

- 点位营业时间、特殊闭店日、最晚入场时间
- 点位预约规则、预约场次、排队画像、最佳游览时间窗
- 餐厅餐段与餐段时间窗
- 酒店入住退房、外宾友好、可订状态、可订日期
- 机场到达/出发缓冲与 terminal 锚点

## 6. 午餐冲突规则（冻结）

- `mealPolicy=auto` 时，系统在 `12:00-14:00` 施加“至少一次用餐事件”约束。
- 若当日已有 `restaurant` 点位且停留时长覆盖午餐窗口，则不再额外插入“午休块”。
- 若无餐厅点位覆盖窗口，则由求解器插入“虚拟午餐事件”（默认 `45` 分钟，可后续配置化）。
- `mealPolicy=off` 时，不施加午餐约束。

## 7. 发布与求解数据门禁

- `isPublished=true` 时，必须满足对应“发布必填”字段。
- 进入求解前，必须满足对应“求解必填”字段；不满足的点位不得进入 `selected candidates`。
- backend 当前门禁额外要求：
  - `spot/shopping` 需具备结构化营业时间；
  - `spot` 需具备 `lastEntryTime`；
  - `restaurant` 需具备 `mealSlots + mealTimeWindowsJson`；
  - `hotel` 需具备 `checkInTime/checkOutTime`，且会过滤 `foreignerFriendly=false` 与 `bookingStatus=sold_out`。
- 任何价格区间字段必须满足 `max >= min`。
