# Optivoy Backend 部署说明

最后更新：2026-03-16

## 生产 Docker 文件检查

- `Dockerfile`：已采用多阶段构建
- 构建阶段：
  - `npm ci`
  - `npm run build`
  - `npm prune --omit=dev`
- 运行阶段：
  - 仅复制 `dist`、`node_modules`、`package.json`
  - 默认启动命令：`node dist/main.js`
- 生产容器可直接执行：
  - `npm run migration:run:prod`
  - `npm run migration:revert:prod`
  - `npm run seed:admin:prod`

## 本地构建架构说明

如果你本地是 Apple Silicon（M1/M2/M3），默认 `docker build` 构建出来的通常是 `linux/arm64` 镜像。

如果服务器是阿里云常见的 `x86_64` 轻量应用服务器，这个镜像不能直接作为生产镜像上传部署。

部署前先确认服务器架构：

```bash
ssh -i ~/.ssh/admin.pem -p <ssh-port> admin@<server-ip> "uname -m"
```

常见结果：

- `x86_64`：生产镜像应构建为 `linux/amd64`
- `aarch64` / `arm64`：生产镜像应构建为 `linux/arm64`

如果你的服务器是常见阿里云 x86 机型，下面所有示例默认使用：

```bash
--platform linux/amd64
```

## 服务器目录约定

建议在服务器上统一使用以下目录：

- `/opt/optivoy/app`
  - 放 `docker-compose.yml`
- `/opt/optivoy/env`
  - 放生产环境变量文件 `.env`
- `/opt/optivoy/releases`
  - 放当前待加载的镜像包
- `/opt/optivoy/backups`
  - 放历史镜像备份，用于回滚

建议先创建目录：

```bash
sudo mkdir -p /opt/optivoy/app /opt/optivoy/env /opt/optivoy/releases /opt/optivoy/backups
sudo chown -R admin:admin /opt/optivoy
chmod 700 /opt/optivoy/env
```

## 首次部署

### 1. 本地构建镜像

```bash
docker buildx build --platform linux/amd64 -t optivoy-backend:prod --load .
docker save optivoy-backend:prod | gzip > optivoy-backend-prod.tar.gz
```

### 2. 上传镜像到 OSS

```bash
ossutil cp optivoy-backend-prod.tar.gz oss://optivoy/docker-images/optivoy-backend-prod.tar.gz
```

### 3. 上传生产环境变量到服务器

推荐继续使用 `scp`：

```bash
scp -i ~/.ssh/admin.pem -P 22 /Users/code-2026/optivoy.top/optivoy-backend/.env.production root@60.205.92.35:/opt/optivoy/env/.env
ssh -i ~/.ssh/admin.pem -p 22 root@60.205.92.35 "chmod 600 /opt/optivoy/env/.env"
```

### 4. 上传 compose 文件到服务器

```bash
scp -i ~/.ssh/admin.pem -P 22 /Users/code-2026/optivoy.top/optivoy-backend/docker-compose.yml root@60.205.92.35:/opt/optivoy/app/docker-compose.yml
```

### 5. 服务器下载镜像包

```bash
ossutil cp oss://optivoy/docker-images/optivoy-backend-prod.tar.gz /opt/optivoy/releases/
```

### 6. 服务器加载镜像

```bash
sudo docker load < /opt/optivoy/releases/optivoy-backend-prod.tar.gz
```

### 7. 执行数据库迁移

```bash
cd /opt/optivoy/app
sudo docker compose run --rm app npm run migration:run:prod
```

### 8. 初始化管理员账号

首次部署且数据库里还没有管理员时执行：

```bash
cd /opt/optivoy/app
sudo docker compose run --rm app npm run seed:admin:prod
```

### 9. 启动容器

```bash
cd /opt/optivoy/app
sudo docker compose up -d
sudo docker compose ps
sudo docker logs --tail 200 optivoy-backend

```

### 10. 查看日志

```bash
sudo docker logs -f optivoy-backend
```

## 升级命令

### 本地

```bash
docker buildx build --platform linux/amd64 -t optivoy-backend:prod --load .
docker save optivoy-backend:prod | gzip > optivoy-backend-prod.tar.gz
ossutil cp optivoy-backend-prod.tar.gz oss://optivoy/docker-images/optivoy-backend-prod.tar.gz
```

### 服务器

先备份当前镜像：

```bash
docker image inspect optivoy-backend:prod >/dev/null 2>&1 && \
docker save optivoy-backend:prod | gzip > /opt/optivoy/backups/optivoy-backend-prod-$(date +%Y%m%d%H%M%S).tar.gz
```

下载并加载新镜像：

```bash
ossutil cp oss://<your-oss-bucket>/docker-images/optivoy-backend-prod.tar.gz /opt/optivoy/releases/
docker load < /opt/optivoy/releases/optivoy-backend-prod.tar.gz
```

执行迁移并重启：

```bash
cd /opt/optivoy/app
docker compose run --rm app npm run migration:run:prod
docker compose up -d
docker compose ps
docker logs --tail 200 optivoy-backend
```

## 本地 Web 联调已部署后端

如果前端在本地 `npm run dev`（如 `http://localhost:5173`）运行，但后端使用阿里云线上服务，需要确认服务端环境变量包含本地来源：

```env
CORS_ORIGIN=https://app.optivoy.top,https://admin.optivoy.top,https://www.optivoy.top,https://optivoy.top,http://localhost:5173,http://localhost:5174
```

修改服务器 `/opt/optivoy/env/.env` 后，重启容器使配置生效：

```bash
cd /opt/optivoy/app
docker compose up -d
docker logs --tail 100 optivoy-backend
```

如果要在本地联调登录/刷新（依赖 HttpOnly refresh cookie），还需要注意：

- `AUTH_REFRESH_COOKIE_SAME_SITE=lax` 时，`localhost` -> `api.optivoy.top` 这类跨站 XHR 通常不会带上 refresh cookie
- 需要跨站带 cookie 时可临时改为 `AUTH_REFRESH_COOKIE_SAME_SITE=none`，并保持 `AUTH_REFRESH_COOKIE_SECURE=true`

## 回滚命令

### 镜像回滚

选择一个历史备份包，例如：

```bash
ls -lh /opt/optivoy/backups
```

加载旧镜像并重新启动：

```bash
docker load < /opt/optivoy/backups/optivoy-backend-prod-YYYYmmddHHMMSS.tar.gz
cd /opt/optivoy/app
docker compose up -d
```

### 数据库回滚

如果本次发布执行过迁移，且迁移与旧镜像不兼容，需要再执行：

```bash
cd /opt/optivoy/app
docker compose run --rm app npm run migration:revert:prod
```

注意：

- 不是所有迁移都适合直接回滚
- 如果迁移包含删字段、删表、数据变换，回滚前必须先确认影响

## 旧 Book Nest 数据清理建议

### RDS

最佳实践：不要复用 `book-nest` 旧数据库，直接新建一个数据库给 `optivoy`。

建议：

- 新建数据库：`optivoy`
- 把 `DB_NAME` 指向新数据库
- 在新数据库里执行 `npm run migration:run:prod`

如果你必须复用原数据库，并且确认里面的 `book-nest` 数据都不要了，可以清空原 schema：

```sql
DROP SCHEMA public CASCADE;
CREATE SCHEMA public;
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
```

执行前提：

- 该数据库没有其他项目共用
- 你确认不再需要 `book-nest` 的任何表和数据

### Tair / Redis

`optivoy-backend` 当前只把 Redis 用于登录限流，不存业务主数据。

最佳实践：

- 如果实例支持逻辑库，优先给 `optivoy` 使用新的 `REDIS_DB`
- 如果实例不支持多逻辑库，且当前实例只给 `book-nest` 使用，可以清空旧库

清空当前逻辑库：

```bash
redis-cli -h <redis-host> -p <redis-port> -a '<redis-password>' -n <redis-db> FLUSHDB
```

如果你清不掉旧库，短期也不会影响 `optivoy` 主业务运行，但建议保持整洁。

## 本地构建常见问题

### 1. 拉取 `node:22-alpine` 超时

如果报错类似：

```text
failed to fetch anonymous token
Get "https://auth.docker.io/token": i/o timeout
```

这不是项目代码问题，而是本地 Docker 无法正常访问 Docker Hub。

可选处理方式：

- 在 Docker Desktop 中配置可用的 registry mirror
- 给 Docker Desktop 配置可用代理
- 先单独执行 `docker pull --platform linux/amd64 node:22-alpine`
- 网络恢复后再重新执行构建

### 2. Mac M3 构建出的镜像架构不对

不要直接用：

```bash
docker build -t optivoy-backend:prod .
```

应显式指定平台，例如：

```bash
docker buildx build --platform linux/amd64 -t optivoy-backend:prod --load .
```
