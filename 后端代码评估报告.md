# 后端代码评估报告（当前状态）

最后更新：2026-03-30

## 1. 产品定位与适配范围

- 目标用户：蒙古国用户（`MN`）赴中国（`CN`）旅游。
- 核心能力：用户在目标中国城市内“自由选点（无顺序）”，系统自动生成最优游玩顺序与酒店方案。
- 当前后端架构：`TripPlanner + transit_cache + OR-Tools optimizer`。
- 当前国家范围约束：位置体系（国家/省/市/机场）仅允许 `CN` 与 `MN`。

## 2. 端到端流程检查结果

### 2.1 Admin 录入阶段（点位与机场）

- 已具备四类点位管理接口：
  - `admin/spots`
  - `admin/shopping`
  - `admin/restaurants`
  - `admin/hotels`
- 机场维护方式：
  - 通过 `admin/locations/airports/import-csv` 导入；
  - 支持 `countries/regions/airports` 数据集导入；
  - 机场导入要求核心字段：`iata_code/name/iso_country/iso_region/municipality`；
  - 若 CSV 含 `latitudedeg/longitudedeg` 会写入坐标。
- 机场导入范围控制：
  - 仅保留 `CN/MN`；
  - 仅保留合法三位大写 IATA 码。

### 2.2 距离矩阵阶段（transit_cache）

- 触发机制：
  - spot/shopping/restaurant/hotel 新增或更新后触发；
  - 机场 CSV 导入后按 touched city 触发。
- 计算范围：
  - 只计算“当前城市（可叠加省份过滤）”内部点位之间的边；
  - 点位集合包含 `spot/shopping/restaurant/hotel/airport`；
  - 仅对已发布且有坐标的点位参与计算。
- 计算内容：
  - 驾车时间（矩阵 API）
  - 步行距离与时间（矩阵 API）
  - 公交通勤时间与路线摘要（方向 API，逐对获取）
- 存储形态：
  - 表 `transit_cache`；
  - 有向边 `fromPointId -> toPointId`；
  - 唯一键：`(city, fromPointId, toPointId)`；
  - 状态字段：`ready/stale/failed`。

### 2.3 用户请求阶段（生成线路）

- 入口接口：`POST /trip-planner/generate`（需登录）。
- 用户只选择：`spot/shopping/restaurant`。
- 酒店不由用户手选，后端自动加载目标城市已发布酒店候选集。
- 机场为可选约束输入（`arrivalAirportCode/departureAirportCode`）。
- 生成前会进行求解门禁：
  - 选中点位必须具备：城市、省份、坐标、建议停留时长；
  - 酒店必须具备：城市、省份、坐标；
  - 若传机场码，对应机场必须存在且有坐标；
  - 选点必须与请求的城市（及可选省份）一致。

### 2.4 求解与返回阶段

- backend 读取 `transit_cache`（仅 `ready`）组装 `distanceMatrix`。
- backend 调用 optimizer `/solve`，传入：
  - 三档强度：`light/standard/compact`
  - 酒店策略：`single/multi`
  - 用餐策略：`auto/off`
  - 到达缓冲：`airportBufferMinutes`（60-120）
- 返回内容包含：
  - 行程天数、每日点位顺序、每日酒店、分段交通信息；
  - 酒店预订链接；
  - 往返机票链接（当前为既有 deep link 规则）。

## 3. 不同点位录入要求（当前实现口径）

### 3.1 通用要求（spot/shopping/restaurant/hotel）

- 多语言名称：`nameZhCN/nameMnMN/nameEn`。
- 城市归属：`province/city`（英文标准名）及对应多语言字段。
- 内容字段：`intro*`、`guide*`。
- 坐标：`latitude/longitude` 建议录入；进入求解时为硬要求。
- 发布控制：`isPublished`（默认 true）。

### 3.2 spot

- 必填：`suggestedDurationMinutes`（30-720）。
- 可选：`placeType`、预约信息、闭馆周几、票价区间。
- 约束：若 `reservationRequired=true`，`reservationUrl` 必须存在。

### 3.3 shopping

- 必填：`suggestedDurationMinutes`（30-720）。
- 可选：`openingHours`、消费区间。

### 3.4 restaurant

- 必填：`suggestedDurationMinutes`（30-720）、`mealSlots`。
- `mealSlots` 枚举：`breakfast/lunch/dinner/night_snack`。
- 可选：`cuisineTags`、预约信息、消费区间、营业时间、休息日。
- 约束：若 `reservationRequired=true`，`reservationUrl` 必须存在。

### 3.5 hotel

- 当前发布字段：名称、归属城市、省份、内容文案；
- 求解硬要求：必须有坐标（无坐标酒店不会进入求解）。
- 可选：星级、外宾友好、入住退房时间、价格区间、`bookingUrl`。

### 3.6 airport

- 来源：CSV 导入。
- 核心字段：IATA 三位码、名称、归属城市。
- 求解硬要求：必须有坐标；无坐标机场码在生成阶段会被拒绝。

## 4. 用户提交数据要求（/trip-planner/generate）

- `city`: 必填，目标中国城市。
- `province`: 可选；若选点无法唯一推断省份则必须提供。
- `arrivalDateTime`: 必填，ISO 时间。
- `arrivalBufferMinutes`: 可选，默认 90，范围 60-120。
- `selectedPointIds`: 必填，UUID 数组，最多 200，去重后不能为空。
- `paceMode`: 必填，`light|standard|compact`。
- `hotelMode`: 必填，`single|multi`。
- `mealPolicy`: 可选，默认 `auto`，可选 `auto|off`。
- `arrivalAirportCode/departureAirportCode`: 可选，传入时必须是目标城市内有效机场码。

## 5. 蒙古国用户场景状态（重点）

- 业务主线正确：蒙古用户来华旅游，线路计算在“中国目标城市内”完成。
- “只算当前城市内矩阵”已满足：矩阵预计算与读取均按 `city` 过滤。
- 乌兰巴托机场（`UBN`）可作为位置体系中的机场基础数据保留。
- 当前 solver 语义是“目标游玩城市到达/离开机场”，不是“蒙古出发地机场”建模。
- 因此：
  - 如果目标是北京行程，`arrivalAirportCode` 应是北京机场（如 `PEK/PKX`）；
  - `UBN` 目前不作为北京行程求解入参。
- 乌兰巴托的 `spot/shopping/restaurant/hotel` 在来华主场景中不是必须数据，可不作为运营重点维护。

## 6. 当前实现边界（需知）

- 若 `transit_cache` 某些边缺失：
  - optimizer 会用坐标进行兜底估算；
  - itinerary 分段展示也会使用 fallback 边（固定保底值）兜底，不会阻断生成。
- `AMAP_ENABLED=false` 时不会进行高德预计算，矩阵依赖会显著下降到兜底路径。
- 机票深链当前维持现状（未切 Travelpayouts）。

## 7. 当前状态结论

- 后端已形成“录入 -> 城内矩阵 -> 用户选点 -> 求解 -> 返回线路与链接”的完整闭环。
- 现阶段最关键的数据质量抓手是：
  - 中国目标城市点位覆盖率；
  - 点位与机场坐标完整性；
  - `transit_cache` 的 ready 覆盖率与新鲜度。
