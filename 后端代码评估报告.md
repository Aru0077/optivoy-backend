## 3. 代码质量评估

### 3.1 架构设计

**优点：**
- Feature-First 模块化清晰，各模块边界明确（module / controller / service / entity / DTO / guard / spec）
- 统一使用 `HttpErrorFilter` 输出标准化错误格式（`{ code, message, details, timestamp, path }`）
- `LoggingInterceptor` 全局记录请求方法、路径、状态码、耗时、IP，便于追踪
- 配置集中管理，通过 `ConfigModule` + Joi 全量校验环境变量，防止配置缺失上线
- 认证三通道（`user-web / user-app / admin-web`）通过 `AuthScope` 守卫隔离，避免 Token 跨通道复用

**不足：**
- Trip Planner Service（`src/modules/trip-planner/trip-planner.service.ts`，65KB）和 Matrix Admin Service（`src/modules/matrix-admin/matrix-admin.service.ts`，57KB）体量过大，应进一步按职责拆分
- 行内注释不足，复杂的距离矩阵组装逻辑缺少说明

### 3.2 DTO 与输入验证

- 全局 `ValidationPipe` 配置：`whitelist: true`（剥离额外字段）、`forbidNonWhitelisted: true`（拒绝未知字段）、`transform: true`（自动类型转换）
- 各模块 DTO 均使用 `class-validator` 完整标注，I18n 字段通过嵌套 DTO 验证
- 边界约束合理（如 `selectedPointIds` 最多 200 个，`paceMode` 枚举限制等）

### 3.3 错误处理

- `HttpErrorFilter` 统一映射 `DEFAULT_ERROR_CODES`，区分 HttpException 与非预期异常
- Amap 客户端实现了完整的 QPS 重试逻辑（`CUQPS_HAS_EXCEEDED_THE_LIMIT` 检测 + 指数退避）
- Redis 连接失败时有优雅降级（开发模式降为内存限流，生产模式显式抛错）
- Optimizer 调用有超时控制（`OPTIMIZER_REQUEST_TIMEOUT_MS`，默认 10s）

### 3.4 国际化（I18n）

- JSONB 字段模式（`nameI18n: { "zh-CN": "...", "mn-MN": "...", "en-US": "..." }`）贯穿所有点位实体
- 通过 `content-i18n.util.ts` 统一处理语言回退逻辑
- 缺点：未使用 `nestjs-i18n` 等框架，I18n 字段格式完全依赖应用层约定，无类型保障

### 3.5 代码一致性

- 各点位实体（Spot / Hotel / Restaurant / Shopping）字段模式高度一致（坐标锚点、营业时间、排队画像、内容文案）
- `BasePlaceService` 抽象了公共 CRUD 逻辑，子类服务继承复用
- 迁移命名语义清晰（如 `AddTransitCacheTable`、`AddSpotEntryExitCoordinates`）

## 5. 安全体系评估

### 5.1 HTTP 层

- Helmet 8.x：CSP、HSTS、X-Frame-Options、Referrer-Policy 全部启用
- CORS：通过 `CORS_ORIGIN` 环境变量严格限制来源
- 全局限流：`ThrottlerGuard`，120 次/60 秒/IP
- 接口级限流：关键端点（矩阵重算、密码重置等）单独设置更严格阈值

### 5.2 认证与授权

- 密码：bcrypt 哈希（`bcrypt@^6.0.0`），不存储明文
- JWT：Access Token（短期）+ Refresh Token（长期）双 Token 方案
- Refresh Token：哈希存储于数据库，非明文
- App 通道独立 Refresh Token（`hashedAppRefreshToken`），防止 Web/App 互相影响
- Admin：2FA（TOTP）强制，Challenge 存于 Redis，有 TTL
- OAuth：Google / Facebook 支持账号关联，防止重复注册
- Token 吊销：`AccessTokenBlacklistService`（Redis）支持登出后立即失效

### 5.3 数据库层

- 所有查询通过 TypeORM QueryBuilder 或 Repository，无 SQL 拼接，防注入
- 敏感字段（`passwordHash`、`hashedRefreshToken`）在 Service 层显式剔除后返回

### 5.4 文件上传

- STS 临时授权（`/uploads/sts`），上传凭证有时效，不暴露主账号 AK/SK
- 阿里云绿网内容审核（图片违规检测）

### 5.5 评级

**安全体系评分：94 / 100**

主要扣分项：
- 未见 SQL Query 日志审计（无法追溯异常查询）
- Admin 权限为单角色 `admin`，无细粒度 RBAC（低优先级）

## 9. 距离矩阵限流问题与最佳实践

### 9.1 问题根因分析

**当前实现文件：**
- `src/modules/transit-cache/amap/amap-transit.client.ts`
- `src/config/amap.config.ts`（`AMAP_TRANSIT_REQUEST_MIN_INTERVAL_MS` 默认 220ms）

**根本问题：**

高德免费 Web API 的实际 QPS 限制：

| 接口 | 实际限制 |
|------|----------|
| `v3/distance`（驾车/步行矩阵）| 每批次 25 起点，QPS ≈ 100/s |
| `v3/direction/transit/integrated`（公交方向）| **QPS = 1/s（有时更低）** |

当前设置 220ms 最小间隔 → QPS ≈ 4.5，**远超公交方向接口的实际限制**，导致频繁触发 `CUQPS_HAS_EXCEEDED_THE_LIMIT`。

**请求量估算：**

| 城市规模 | 点位数 | Transit 边数 | 按 1 QPS 最少耗时 |
|----------|--------|--------------|-------------------|
| 小城市 | 10 个 | 90 条 | 1.5 分钟 |
| 中等城市 | 20 个 | 380 条 | 6.3 分钟 |
| 大城市（北京） | 50 个 | 2450 条 | 41 分钟 |

频繁限流会使实际耗时数倍于上表，且作业状态被标记为 `rate_limited`。

---

### 9.2 最佳实践方案

#### 方案 A：令牌桶限流（立即可改，核心修复）

替换 `amap-transit.client.ts` 中的最小间隔机制，实现真正的令牌桶：

```typescript
// 在 AmapTransitClient 中添加
private tokenBucket = { tokens: 1, lastRefillAt: Date.now() };
private readonly TRANSIT_QPS = 1; // 高德公交方向 QPS 上限

private async acquireTransitToken(): Promise<void> {
  const now = Date.now();
  const elapsed = (now - this.tokenBucket.lastRefillAt) / 1000;
  this.tokenBucket.tokens = Math.min(
    1,
    this.tokenBucket.tokens + elapsed * this.TRANSIT_QPS,
  );
  this.tokenBucket.lastRefillAt = now;

  if (this.tokenBucket.tokens >= 1) {
    this.tokenBucket.tokens -= 1;
    return;
  }

  // 精确计算等待时间
  const waitMs = Math.ceil((1 - this.tokenBucket.tokens) / this.TRANSIT_QPS * 1000);
  await new Promise(resolve => setTimeout(resolve, waitMs));
  this.tokenBucket.tokens = 0;

}
```

#### 方案 B：驾车/步行矩阵批量优化（立即可改，减少 API 调用量）

#### 方案 C：指数退避策略（立即可改，提高重试成功率）

当前退避：`transitQpsBackoffMs * retryCount`（线性）

**改为指数退避：**

```typescript
// amap-transit.client.ts 中的 QPS 重试逻辑
private async withTransitQpsRetry<T>(fn: () => Promise<T>): Promise<T | null> {
  const backoffs = [1000, 2000, 4000]; // 1s, 2s, 4s
  for (let attempt = 0; attempt <= backoffs.length; attempt++) {
    const result = await fn();
    if (!this.isQpsExceeded(result)) return result;
    if (attempt < backoffs.length) {
      this.logger.warn(`Transit QPS exceeded, retry ${attempt + 1} after ${backoffs[attempt]}ms`);
      await this.delay(backoffs[attempt]);
    }
  }
  this.logger.error('Transit QPS exceeded after all retries, skipping edge');
  return null;
}
```

#### 方案 D：智能边过滤（减少请求量 30-50%）

不是所有 N*(N-1) 条边都需要公交数据：

```typescript
// transit-cache-precompute.service.ts
function shouldComputeTransit(from: CityPoint, to: CityPoint): boolean {
  // 1. 酒店→酒店 无需公交（用户只住一家）
  if (from.type === 'hotel' && to.type === 'hotel') return false;

  // 2. 机场→机场 无需公交
  if (from.type === 'airport' && to.type === 'airport') return false;

  // 3. 直线距离 > 80km，公交数据可靠性低（通常同城不会超过）
  const distKm = haversineDistance(from, to);
  if (distKm > 80) return false;

  return true;
}

// 使用 haversine 公式（不依赖 API）做前置过滤
function haversineDistance(a: { lat: number; lng: number }, b: { lat: number; lng: number }): number {
  const R = 6371;
  const dLat = ((b.lat - a.lat) * Math.PI) / 180;
  const dLng = ((b.lng - a.lng) * Math.PI) / 180;
  const h =
    Math.sin(dLat / 2) ** 2 +
    Math.cos((a.lat * Math.PI) / 180) * Math.cos((b.lat * Math.PI) / 180) * Math.sin(dLng / 2) ** 2;
  return R * 2 * Math.asin(Math.sqrt(h));
}
```

#### 方案 E：BullMQ 持久化队列（中期建议，生产级方案）

对于城市级全量重算，当前单进程异步模式存在以下风险：
- 服务重启丢失进度（需重新从头计算）
- 无法在多实例部署中协调任务
- 无法可视化任务进度

**推荐架构：**

```
POST /admin/matrix/recompute-city
    └─ 将每条边拆分为独立 Job 入队（BullMQ）
         └─ transit-matrix-queue
              ├─ limiter: { max: 1, duration: 1000 }  // Transit: 1 job/s
              ├─ 失败自动重试（指数退避）
              ├─ 进度持久化于 Redis
              └─ Bull Board 可视化监控

driving-matrix-queue
    └─ limiter: { max: 3, duration: 1000 }  // 驾车矩阵可更高并发
```

**安装：**
```bash
npm install bullmq @nestjs/bullmq
# Bull Board 可选
npm install @bull-board/nestjs @bull-board/api @bull-board/express
```

**优点：**
- 服务重启后自动从 Redis 恢复未完成的 Job
- 内置限流（`limiter`）与重试（`attempts`）
- 支持优先级队列（新增点位的邻域计算优先于全量重算）
- 可视化 Bull Board 监控页面
